# Monte Carlo Polymer Simulation

This repository contains a Fortran-based Monte Carlo simulation of a polymer chain.
The project uses a `Makefile` that handles compiling, running, dependency management, and job submission to the university HPC cluster.

## Prerequisites
* **Fortran Compiler:** `gfortran`
* **Python 3:** For data visualization.
* **Make:** To run the automated pipeline.

## Local Usage

For testing, debugging, and short simulations, you can run the entire pipeline on your local machine.

### 1. Run the Simulation
To compile the Fortran code (if modified) and run the simulation locally, use:
```bash
make run
```

### 2. Generate Plots

To generate visualizations of the simulation data:
```bash
make plot
```

This command will:
1. Run the simulation (if the data is missing).
2. Create an isolated Python virtual environment (.venv).
3. Install the required dependencies (numpy, matplotlib).
4. Generate the plots and save them to results/plots/.

### 3. Cleanup
```bash
make clean      # Deletes data and build folders
make clean-all  # Deletes data, build folders, and the Python .venv
```

## Cluster Usage (High-Performance Computing)
For long production runs (e.g., $10^8$ steps), the simulation should be offloaded to the university cluster (cerqt2.qt.ub.edu). The Makefile fully automates the syncing, submission, and retrieval of these jobs.

### Step 1: Passwordless SSH Setup (One-Time)
To allow the Makefile to communicate with the cluster without prompting for a password every time, you must set up an SSH key pair.

#### 1. Generate the key locally:
```bash
ssh-keygen -t ed25519
```
#### 2. Copy the key to the cluster:
```
ssh-copy-id your_username@cerqt2.qt.ub.edu
```

### Step 2: Configure the. env File
For security, your cluster credentials are never hardcoded or uploaded to GitHub. You must create a file named. env in the root of the project.

Create a .env file with the following variables:
```bash
CLUSTER_USER=your_ub_username
CLUSTER_HOST=cerqt2.qt.ub.edu
CLUSTER_DIR=~/EIA_project
CLUSTER_EMAIL=your_email@ub.edu
```

### Step 3: Cluster Workflow Commands
Once your SSH keys and .env file are set up, you can control the cluster entirely from your local terminal.

#### Submit a Job:
```bash
make run-cluster
```

#### Check Job Status:
```bash
make check-cluster
```

#### Fetch and Plot Results:
```bash
make plot-cluster
```
Once you receive the completion email, run this command. It will download the generated data from the cluster to your local results/cluster_archive/ folder, locate the newest run, and generate the plots locally using your. venv.

#### Clean Cluster Workspace:
```bash
make clean-cluster
```

## The Essence of the Monte Carlo Simulation:

Regardless of the kind of simulation, at its core, what the program does is:

1. Initializes the molecule with a given distribution. The initial conditions might be relevant, especially for shorter simulations.
2. For each _MC step_, it tries to change the current state of the system 
    * Proposing a change $\phi_k \to \phi_k + \delta\phi$ for dihedral $1\leq\phi\leq N-1$.
    * Compute the associated energy change $\Delta E$.
    * Accept or reject the change with relative probability $\min(1, e^{-\Delta E/T})$ (Metropolis rule).
    * Repeat for a total number of attempts per MC step (ideally 1 attempts per degree of freedom).
3. Sample the relevant observables (energy, end-to-end distance, ...) at a given sampling frequency.
4. Repeat for a total of number of _equilibration_ and _production_ steps.

## Possible Modes of Simulating:

The program allows to perform different kinds of simulations, independently or combined. In short, it can simulate:

1. In **serial** or **parallel** (using `MPI`).
2. From a **fully stretched** or a **random dihedrals** initial configurations.
3. With or without **Verlet lists**.
4. With **replica parallelism**.
5. With **replica exchange** (requires replica parallelism).
6. With **energy loop** parallelism.

### How to choose a simulation 'mode':

To specify the mode of simulation, edit the `input.dat` file appropriately, and specify the corresponding keywords when compiling/running the code via `make`. 

1.  **Serial vs. Parallel**: The `Makefile` has the sequential mode by default. To override this, choose parallel simulation. Write:
    ```bash
    make MODE=parallel
    ```

2. **Initial configuration**: In `input.dat`, the `initRandom` boolean variable controls if the initial configuration is fully stretched or is constructed by sampling uniformly random dihedrals. Set it to `0` for a fully stretched chain or to `1` for a random dihedrals configuration.
   
3. **Verlet lists**: In `input.dat`, the `isVlist` boolean variable controls if the simulation is done using Verlet lists (set it to `1`) or not (set it to `0`).

4. **Replica parallelism**: To simulate in parallel with more than one replica, specify the number of cores to use in the `Makefile`. By default it is set to 4. Make sure to also specify that the simulation is in parallel. Write:
    ```bash
    make MODE=parallel NCORES=16
    ```

5. **Replica exchange**: Assuming replica parallelism, to implement replica exchange (i.e., parallel tempering), in `input.dat` the variable `N_SWAP` controls the frequency at which the replica exchange takes place. Set it to `0` for no replica exchange at all or to a *positive integer* otherwise. Also, the variable `MAX_TEMP` controls the maximum available temperature for the replicas.
   
6. **Energy loop parallelism**: In `input.dat`, the `nproc_per_replica` variable sets the number of processors designated to each replica to compute the energy loops. 

**Note**: the `MODE` and `NCORES` `Makefile` specifications work in combination with the ones explained previously. For instance to run a simulation in the cluster with replica parallelism one would just write `make run-cluster MODE=parallel NCORES=8`. 

**Note**: To make a simulation *only* with replica parallelism (i.e., *without* replica exchange) make sure to set `N_SWAP` to `0` also.
