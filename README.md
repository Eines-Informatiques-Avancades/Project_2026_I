# Monte Carlo Polymer Simulation

This repository contains a Fortran-based Monte Carlo simulation of a polymer chain. The project uses a `Makefile` pipeline that handles compiling, executing, Python-based data post-processing, dependency management, and job submissions to HPC clusters (Slurm and SGE).

## 1. The Essence of the Monte Carlo Simulation

Regardless of the kind of simulation (serial, parallel, or grid sweep), at its core, the underlying physics engine performs the following:

1. Initializes the molecule with a given distribution (e.g., fully stretched or random dihedrals). Initial conditions strongly influence shorter equilibration periods.
2. For each *MC step*, it tries to change the current state of the system:
    * Proposes a change $\phi_k \to \phi_k + \delta\phi$ for a dihedral $1\leq\phi\leq N-1$.
    * Computes the associated energy change $\Delta E$.
    * Accepts or rejects the change with relative probability $\min(1, e^{-\Delta E/T})$ (Metropolis rule).
    * Repeats for a total number of attempts per MC step (ideally 1 attempt per degree of freedom).
3. Samples relevant observables (energy, end-to-end distance, etc.) at a given sampling frequency.
4. Repeats for a total number of *equilibration* and *production* steps.

## 2. Prerequisites & Environment Setup

To compile and run this code locally, ensure the following dependecies are installed:

* **Fortran Compiler:** `gfortran` (for sequential codebase)
* **MPI Library:** `openmpi-bin` and `libopenmpi-dev` (required, including `mpifort`, for parallel/replica simulations)
* **Python 3:** For plotting, data visualization, and managing the built-in virtual environment (`.venv`).
* **Make:** To run the automated pipeline natively.
* **SSH:** For connecting to remote machines (clusters).
* **rsync:** For synchronizing files between local and remote machines (clusters).

### Apptainer / Singularity (Optional but Recommended)

If you require a completely reproducible, non-OS-invasive installation without manually installing Fortran, MPI, and Python locally, this project provides an `apptainer.def` container definition.

To build the container image (`md_tools.sif`):
```bash
apptainer build md_tools.sif apptainer.def
```
Once built, you can execute Makefile commands inside the container without polluting your host OS. For example:
```bash
apptainer exec md_tools.sif make MODE=sequential plot
```

## 4. Simulation Configuration & Modes

The core simulation is controlled by the `input.dat` file, which provides options for several modes:

1. **Initial configuration**: `initRandom` set to `0` starts fully stretched, `1` starts with randomly sampled dihedrals.
2. **Verlet lists**: `isVlist` controls if neighbor lists speed up non-bonded energy calculations (`1` for yes, `0` for no).
3. **Serial vs. Parallel (MPI)**: `Makefile` targets the `parallel` version by default. To run serially, use `make MODE=sequential`.
4. **Replica parallelism**: Controlled by the invocation flag `make NCORES=N` along with `MODE=parallel`.
5. **Replica exchange**: For replica exchange, set `N_SWAP > 0` in `input.dat` (freq. of exchange). `MAX_TEMP` determines the maximum boundary. If `N_SWAP == 0` but the simulation is running in parallel, it will run as a standard replica parallel simulation without replica exchange.
6. **Energy loop parallelism**: Set `nproc_per_replica` inside `input.dat` to split atomic iterations along multiple ranks.

**Example combination**: Running an 8-core replica computation:
```bash
make run MODE=parallel NCORES=8
```

## 5. Local Usage

For testing, debugging, and visualization, you can run the pipeline directly on your personal machine.

### 1. Run the Simulation
```bash
make run
```
*Tip: Override modes using variables: `make run MODE=sequential`.*

### 2. Generate Plots
```bash
make plot
```
This automatically runs the simulation (if no recent data exists), creates an isolated `.venv`, installs the required packages from `requirements.txt`, and generates visuals in `results/plots/`.

### 3. Archive Results
```bash
make snapshot
# or
make snapshot my_snapshot
```

### 4. Cleanup
```bash
make clean      # Deletes data and build folders
make clean-all  # Deletes data, build folders, and the Python virtual environment
```

## 6. High-Performance Computing (HPC) Workflows

For production (e.g., $10^8$ steps), offload to a cluster. The repository supports two schedulers: Slurm and Sun Grid Engine (SGE).

### Using SGE (e.g., cerqt2.qt.ub.edu)

The repository uses `Makefile` macros to automate cluster execution, including synchronizing via rsync and batch job submission. Add your SGE remote details into a `.env` file first:
```bash
CLUSTER_USER=your_ub_username
CLUSTER_HOST=cerqt2.qt.ub.edu
CLUSTER_DIR=~/EIA_project
CLUSTER_EMAIL=your_email@ub.edu
```

*Note: For a best experience, make sure you have a Passwordless SSH Setup. See [Appendix A: Passwordless SSH Setup](#appendix-a-passwordless-ssh-setup) for instructions.*

Then use the following automated pipeline commands from your local machine:
1. **Submit Job:** `make run-cluster`
2. **Check Status:** `make check-cluster`
3. **Fetch & Plot:** `make plot-cluster`
4. **Clean Remote Workspace:** `make clean-cluster`

### Using Slurm

The repository provides scripts for execution on compute nodes using Apptainer containers to guarantee compatibility. Ensure you have built `md_tools.sif` locally on the head node first.

**Standard Run (1 Configuration):**
```bash
sbatch submit_slurm.sh
```

**Parameter Grid Sweep (Job Arrays):**
To sweep through a matrix of variables (e.g., 6 temperatures x 4 dihedral max angles = 24 isolated runs), use the array script:
```bash
sbatch submit_slurm_array.sh
```
*Slurm handles spinning up independent node tasks, patching their `input.dat` files automatically, and depositing their outputs into isolated run folders.*

## Appendix A: Passwordless SSH Setup

To allow the Makefile to communicate with the cluster without prompting for a password every time, you must set up an SSH key pair.

#### 1. Generate the key locally:
```bash
ssh-keygen -t ed25519
```
*(Press Enter to accept the default file location and leave the passphrase empty if you want a fully automated execution.)*

#### 2. Copy the key to the cluster:
```bash
ssh-copy-id your_username@cerqt2.qt.ub.edu
```
